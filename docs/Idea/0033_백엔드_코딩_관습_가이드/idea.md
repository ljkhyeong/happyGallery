# 백엔드 코딩 관습 가이드

> 흔한 실수와 모범 사례를 누적하는 엔지니어링 가이드.
> 새로운 항목이 생기면 해당 섹션에 추가한다.

---

## 1. 보안

### 1-1. IDOR 방지 — 리소스 소유권 검증

**흔한 실수:** 인증(Authentication)만 확인하고 인가(Authorization)를 빠뜨린다. `PUT /api/orders/5` 요청에서 로그인 여부만 보고 수정을 허용하면, 공격자가 ID를 바꿔 남의 리소스를 조작할 수 있다.

**모범 사례:**
- 리소스를 조회한 뒤 `ownerId == 현재 userId`를 반드시 검증한다.
- 불일치 시 `NotFoundException`을 던져 리소스 존재 여부 자체를 노출하지 않는다.

**현재 적용 상태:** 모든 회원 API에서 서비스 계층 소유권 검증 적용 완료. 게스트 API는 토큰 해시 비교로 보호.

## 2. 성능

### 2-1. 컬렉션 선택 — List vs HashSet

**흔한 실수:** 금지어 목록, 권한 목록처럼 수천~수만 개의 문자열을 `List<String>`에 담고 `list.contains(word)`로 검사한다. `List`는 O(N) 선형 탐색이라 데이터가 많을수록 CPU를 소모한다.

**모범 사례:**
- 포함 여부(`contains`)를 자주 확인하는 컬렉션은 `HashSet`을 사용한다.
- 해시 기반 O(1) 탐색으로 데이터 규모에 무관하게 일정한 성능을 보장한다.

**현재 적용 상태:** `List.contains()`를 대량 데이터 lookup에 사용하는 코드 없음.

## 3. JPA / 영속성

### 3-1. save() vs saveAndFlush()

**흔한 실수:** `save()` 직후 DB 제약 조건 위반을 `try-catch`로 잡으려 하지만, 쿼리가 트랜잭션 커밋 시점에 날아가므로 예외가 잡히지 않는다.

**모범 사례:**
- 일반 비즈니스 로직에서는 `save()`를 사용한다 (쓰기 지연으로 성능 최적화).
- DB 레벨 에러를 즉시 감지해야 하거나, 트리거/프로시저 결과를 바로 확인해야 할 때만 `saveAndFlush()`를 사용한다.
- 제약 조건 위반은 가능한 한 애플리케이션 레벨에서 사전 검증한다 (도메인 가드 메서드).

**현재 적용 상태:** 모든 비즈니스 로직이 `save()` 사용. `saveAndFlush()`는 배치 즉시 반영이 필요한 `OrderAutoRefundProcessor` 1건만 사용.
