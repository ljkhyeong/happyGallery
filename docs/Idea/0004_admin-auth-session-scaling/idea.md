# 관리자 인증 세션 확장 검토 메모

**날짜**: 2026-03-17  
**상태**: Backlog

---

## 배경

현재 관리자 인증은 UUID 기반 Bearer 세션 토큰과 서버 메모리 저장소를 사용한다.  
단일 인스턴스 운영에는 단순하고 충분하다. 다만 수평 확장이나 강한 세션 제어가 필요해지면 다시 검토해야 한다.

PRD에는 현재 계약만 남기고, 확장 옵션 비교는 이 문서에서 관리한다.

---

## 현재 기준

- 로그인 성공 시 UUID 세션 토큰 발급
- `Authorization: Bearer {token}` 헤더 사용
- 세션 만료 8시간
- 단일 인스턴스 기준 서버 메모리 저장소
- 로컬/테스트에서는 `X-Admin-Key` 폴백 허용 가능

---

## 재검토 트리거

아래 중 하나라도 발생하면 인증 저장소 전략을 다시 검토한다.

1. 운영 인스턴스를 2대 이상으로 확장해야 한다.
2. 전체 로그아웃, 강제 세션 폐기, 세션 조회 같은 운영 요구가 커진다.
3. 장애 시 세션 유실 허용 범위를 낮춰야 한다.

---

## 후보안

### 1. 공유 세션 저장소

- 예: Redis 기반 세션 저장소
- 장점: 강제 만료와 세션 조회가 쉽다.
- 단점: 운영 의존성이 늘어난다.

### 2. JWT 기반 토큰

- 장점: 서버 저장소 의존도를 줄일 수 있다.
- 단점: 즉시 폐기와 권한 회수 정책이 더 복잡해진다.

### 3. Opaque Token + 공유 저장소

- 장점: 현재 Bearer UX를 유지하면서 세션 제어를 강화할 수 있다.
- 단점: 저장소 운영 비용은 그대로 든다.

---

## 비교 기준

- 강제 로그아웃/권한 회수 용이성
- 운영 복잡도
- 감사 추적 및 세션 조회 가능성
- 장애 시 복구 단순성

---

## 현재 결론

지금은 서버 메모리 세션 유지가 가장 단순하다.  
수평 확장이 필요해지면 Redis 기반 공유 세션과 JWT를 비교해 다시 결정한다.
