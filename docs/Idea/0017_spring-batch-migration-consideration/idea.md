# Spring Batch 마이그레이션 검토

**날짜**: 2026-03-19
**상태**: 보류

---

## 현재 구조

현재 배치 기능은 Spring Batch 없이 아래 조합으로 운영한다.

- `BatchScheduler`: `@Scheduled` 기반 스케줄 진입점
- `BatchExecutor`: 후보 조회 → 건별 처리 → 실패 격리 → `BatchResult` 집계
- `BatchLoggingAspect` + `@BatchJob`: 공통 시작/완료/실패 로그와 `batch-*` requestId 주입
- admin 수동 트리거 API: `POST /admin/orders/expire-pickups`, `POST /admin/passes/expire`

주요 작업은 다음과 같다.

- 주문 승인 SLA 초과 자동환불
- 픽업 만료 자동취소·환불
- 8회권 만료 처리
- 8회권 만료 예정 알림
- 예약 D-1 / 당일 리마인드

주문 자동환불, 픽업 만료 같은 상태 전이 배치는 건별 `REQUIRES_NEW` 트랜잭션과 낙관적 락 재시도를 사용한다.

## 지금 당장 Spring Batch가 필요하지 않은 이유

### 1. 현재 운영 요구를 이미 대부분 충족한다

- 스케줄 실행 시각은 `BatchScheduler` 한 곳에서 관리된다.
- 성공/실패 건수와 실패 사유는 `BatchResult`로 표준화되어 있다.
- 수동 트리거와 정기 실행이 같은 서비스 경계를 공유한다.
- 주문/픽업 만료 배치는 건별 실패 격리와 재시도까지 이미 구현돼 있다.

### 2. 현재 배치는 ETL보다 도메인 상태 전이에 가깝다

현재 작업은 대용량 파일 처리나 다단계 데이터 파이프라인보다,
"상태 조건을 만족하는 엔티티를 찾아 정책을 실행한다"는 성격이 강하다.

- 주문 상태 전이 + 환불
- 픽업 만료 + 재고 복구
- 8회권 만료 + ledger 기록
- 예약 리마인드 발송

즉, Step/Chunk 모델보다 도메인 서비스 호출이 중심이다.

### 3. 마이그레이션 비용이 작지 않다

Spring Batch로 옮기면 아래 항목도 함께 바꿔야 한다.

- `spring-boot-starter-batch` 도입
- JobRepository 및 메타데이터 테이블 운영
- 기존 수동 트리거 API의 `JobLauncher` 연동 방식 재정의
- 기존 `BatchResult` 계약과 운영 로그 규약 정렬
- 통합 테스트와 운영 문서 재작성

지금 문제가 분명하지 않은 상태에서 옮기면 프레임워크 복잡도만 늘어날 가능성이 크다.

## 먼저 보강할 항목

Spring Batch 전환보다 우선순위가 높은 항목은 다음과 같다.

### 1. 예약 리마인드 배치의 실패 격리 보강

`BookingReminderBatchService`는 현재 다른 배치처럼 공통 실행기 기반 건별 실패 집계를 사용하지 않는다.

- 한 건 알림 예외가 전체 실행을 끊지 않도록 구조를 맞출 필요가 있다.
- 필요하면 중복 발송 체크도 함께 검토

### 2. 대량 처리 대비 조회 방식 보강

현재 배치들은 대상을 `List`로 한 번에 조회하는 패턴이 많다.

- 페이지네이션
- 커서 기반 반복
- 처리 건수 상한/윈도우 분할

같은 식의 보강이 먼저 가능하다.

### 3. 다중 인스턴스 중복 실행 방지

Spring Batch 자체만으로는 다중 인스턴스 환경의 스케줄 중복 실행 문제가 자동 해결되지 않는다.
필요하면 다음을 별도로 검토한다.

- ShedLock
- 외부 스케줄러
- leader election / 단일 실행 인스턴스 정책

## Spring Batch 재검토 조건

아래 조건 중 하나 이상이 현실화되면 Spring Batch 재검토 우선순위를 올린다.

### A. 재시작 가능성이 운영 요구가 되는 경우

- 배치가 수천~수만 건 이상 처리됨
- 중간 실패 후 "처리 중단 지점부터 재시작"이 중요해짐
- 장시간 실행 Job의 상태 추적이 필요해짐

### B. Step 기반 파이프라인이 필요한 경우

- 파일 import/export
- 외부 시스템 적재
- 전처리 → 검증 → 저장 → 후처리 같은 다단계 흐름

### C. 운영 메타데이터 요구가 커지는 경우

- Job/Step 실행 이력 조회
- 재실행 정책
- 배치 파라미터 이력 관리
- 운영자가 실행 기록을 화면에서 직접 추적해야 함

## 잠정 결론

지금은 **Spring Batch로 전면 전환하기보다 현재 커스텀 배치를 유지하고 보강하는 편이 더 합리적**이다.

우선순위는 다음과 같다.

1. 예약 리마인드 배치 실패 격리/중복 방지 보강
2. 대량 처리 대비 페이지네이션/윈도우 처리 검토
3. 다중 인스턴스 중복 실행 방지 전략 검토
4. 위 조건이 현실화되면 Spring Batch 도입 재평가
