# BFF(Backend For Frontend) 패턴 검토

> 클라이언트 종류가 다양해질 때 각 프론트엔드 전용 백엔드를 두어 오버패칭을 줄이는 패턴을 검토한다.

---

## 배경

모바일 앱, 웹 브라우저, 스마트워치 등 프론트엔드(클라이언트)의 종류가 다양해지면서 등장한 패턴이다.
단일 API 게이트웨이나 공통 API 하나로 모든 클라이언트를 대응하면, 모바일에서는 필요 없는 대량의 데이터까지 억지로 받아야 하는 **오버패칭(Over-fetching)** 문제가 생긴다.

BFF는 각 클라이언트 전용 맞춤형 백엔드(예: Web용 BFF 서버, Mobile용 BFF 서버)를 앞단에 두어, 해당 프론트엔드 화면에 딱 맞는 데이터만 조립하고 필터링해서 내려주는 역할을 한다.

---

## 핵심 개념

| 항목 | 설명 |
|------|------|
| 오버패칭 해소 | 클라이언트별로 필요한 필드만 선별하여 응답 크기를 줄인다 |
| 관심사 분리 | 프론트엔드 요구사항 변경이 공통 API에 영향을 주지 않는다 |
| 독립 배포 | 각 BFF를 별도로 배포·스케일링할 수 있다 |
| 트레이드오프 | BFF 서버 수만큼 운영 복잡도가 증가한다 |

---

## happyGallery 적용 가능성

- 현재 클라이언트는 **웹 브라우저 단일**이므로 BFF를 도입할 실질적 필요는 없다.
- 향후 모바일 앱이나 키오스크 등 클라이언트가 추가되는 시점에 재검토한다.
- 도입 시 기존 `/api/v1/` 엔드포인트는 내부 서비스 API로 유지하고, BFF 레이어가 이를 조합하는 구조를 고려한다.
