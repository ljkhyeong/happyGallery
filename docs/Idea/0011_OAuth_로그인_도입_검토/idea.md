# OAuth 로그인 도입 검토 메모

**날짜**: 2026-03-17  
**상태**: 검토 메모 (도입 전 검토)

---

## 배경

현재 회원 인증은 이메일/비밀번호 + 세션 쿠키 기반으로 동작한다.  
여기에 OAuth를 추가하면 로그인 진입 장벽은 낮아질 수 있지만, 이 프로젝트에서는 단순한 로그인 버튼 추가보다 계정 연결 정책이 더 큰 설계 포인트다.

특히 비회원 주문/예약/8회권과 `/my`의 비회원 이력 가져오기(claim) 흐름이 이미 존재하므로, OAuth는 인증 수단 추가와 동시에 기존 회원 계정 연결 규칙을 정의해야 한다.

---

## 기대 효과

- 회원가입/로그인 진입 장벽 완화
- 비밀번호 분실/재설정 UX 부담 감소
- 모바일 브라우저 기준 빠른 로그인 경험 제공

---

## 핵심 리스크

### 1. 기존 계정 연결 정책

현재:
- 현재 회원 계정의 기준 식별자는 이메일, 세션, 전화번호 검증 흐름이다.

문제:
- 기존 이메일 회원이 나중에 Google로 로그인하면 같은 계정으로 합칠지 새 계정으로 만들지 정책이 필요하다.
- OAuth 제공자가 제공하는 이메일이 변경되거나 비공개일 수 있다.

### 2. 비회원 이력 가져오기(claim) 흐름과의 충돌

현재:
- 비회원 이력 이전은 `/my`에서 전화번호 재인증을 통해 진행한다.

문제:
- OAuth 로그인만으로는 전화번호 소유 증명이 되지 않는다.
- 제공자 프로필 정보만으로 비회원 이력을 자동으로 가져오게 하면 오인 연결 위험이 있다.

### 3. 온보딩 복잡도 증가

현재:
- 현재는 로그인/회원가입 후 `redirect`, `claim`, 전화번호 미리 채우기 흐름이 비교적 단순하다.

문제:
- OAuth 첫 로그인 시 이름/이메일만 있고 전화번호가 없을 수 있다.
- 이 경우 추가 전화번호 입력, 재인증, claim 안내까지 이어지는 후속 안내 흐름을 설계해야 한다.

### 4. 운영/보안 고려

- 제공자 장애 시 로그인 대체 경로가 필요하다.
- 제공자별 callback, secret, state/redirect 검증이 필요하다.
- 관리자 인증에는 섞지 않고 회원 인증 전용으로 한정해야 한다.

---

## 권장 접근

1. 자체 이메일/비밀번호 로그인을 유지한다.
2. OAuth는 회원 로그인 수단 추가로만 도입하고, 비회원 이력 가져오기(claim)의 대체 수단으로 보지 않는다.
3. 첫 도입은 Google 단일 제공자로 제한한다.
4. OAuth 첫 로그인 후에도 전화번호가 없으면 현재의 phone verification onboarding을 그대로 태운다.
5. 비회원 이력 가져오기(claim)는 계속 전화번호 재인증 기반으로 유지한다.
6. 기존 이메일 회원과 OAuth 계정 연결 정책을 PRD/ADR 수준에서 먼저 확정한 뒤 구현한다.

---

## 추천 도입 순서

### 1단계

- Google OAuth 로그인만 추가
- 신규 회원 생성 또는 기존 이메일 계정 연결 정책 확정
- `/login`, `/signup` UI에 보조 로그인 수단으로 노출

### 2단계

- OAuth 첫 로그인 후 전화번호가 없는 사용자를 위한 추가 인증 안내 흐름
- `/my?claim=1` 흐름과 결합한 post-login 안내 정리

### 3단계

- 실제 가입 전환율, 로그인 성공률, claim 완료율을 보고 확대 여부 판단

---

## 현재 결론

OAuth는 가치가 있지만 지금 프로젝트에서는 “로그인 편의 기능”보다 “계정 연결 정책” 과제에 가깝다.  
따라서 바로 구현하기보다, 기존 이메일 회원과 비회원 이력 가져오기(claim) 흐름을 해치지 않는 연결 규칙을 먼저 정의한 뒤 Google 단일 provider부터 검토하는 것이 맞다.
