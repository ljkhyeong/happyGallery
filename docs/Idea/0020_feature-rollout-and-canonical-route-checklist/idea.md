# 기능 롤아웃 및 표준 경로 체크리스트

**날짜**: 2026-03-20  
**상태**: 검토 중

---

## 배경

공개 경로와 회원/비회원 전환 흐름은 구현만 끝난다고 정리되지 않는다.
문서, E2E, 운영 카피, alias 정리 시점까지 같이 맞춰야 전체 비용이 줄어든다.

하지만 실제 작업에서는 표준 경로나 최소 운영 지표, 문서 반영이 기능 완료 뒤에 따라붙는 경우가 반복됐다.

그 결과:

- README/PRD/E2E 동기화 커밋이 뒤늦게 반복된다.
- alias를 얼마나 유지할지, 언제 제거할지 기준이 늦게 정해진다.
- 운영에서는 새 흐름이 들어왔는데 최소 metric/log가 빠진 채 배포될 수 있다.

---

## 문제 정의

### 1. 표준 경로가 구현 뒤에야 정해진다

As-Is:
- 새 공개 경로를 연 뒤에야 표준 경로와 legacy alias 정리 여부를 논의한다.

To-Be:
- 기능 설계 단계에서 표준 경로, 필요한 alias, alias 종료 시점을 같이 기록한다.

### 2. 사용자 흐름 문서와 브라우저 검증이 뒤늦게 맞춰진다

As-Is:
- README, PRD, Playwright 핵심 브라우저 테스트가 구현 후속 커밋에서 맞춰지는 일이 잦다.

To-Be:
- 새 공개 흐름은 구현 PR 안에서 문서와 E2E 반영 범위를 함께 체크한다.

### 3. 최소 운영 가시성 기준이 기능마다 들쑥날쑥하다

As-Is:
- 기능은 완료됐지만 requestId 로그, 핵심 metric, 실패 관찰 포인트가 나중에 붙는 경우가 있다.

To-Be:
- 신규 핵심 흐름에는 최소한의 로그/metric/doc 반영 여부를 feature 정의 단계에서 같이 적는다.

---

## 제안

새 공개 경로나 핵심 사용자 흐름을 추가할 때 아래 체크리스트를 같이 기록한다.

1. 표준 경로는 무엇인가
2. alias가 필요한가, 필요하면 언제 제거할 것인가
3. README / PRD / API 계약 / E2E 중 무엇을 같이 갱신해야 하는가
4. requestId 로그, 핵심 metric, 운영 대시보드/알림 포인트가 필요한가
5. guest/member 전환이나 claim 흐름과 어떤 연결 규칙을 가지는가
