# Bulkhead(격벽) 패턴 도입 검토 메모

**날짜**: 2026-03-06  
**상태**: Backlog (추후 검토)

---

## 배경

외부 결제/알림 API가 느려지면 요청 처리 스레드가 오래 점유될 수 있다.
그러면 관련 없는 조회 API까지 같이 느려지거나 장애 영향을 받을 수 있다.

---

## 문제 시나리오

1. 결제/환불 API 응답 지연
2. 톰캣 요청 스레드 점유 증가
3. 일반 조회/예약 API까지 대기 증가
4. 전체 서비스 체감 장애

---

## 검토 방향

Resilience4j를 사용해 외부 연동 경로를 격리한다.

1. `Timeout`으로 장기 점유 차단
2. `CircuitBreaker`로 장애 연쇄 차단
3. `Bulkhead`로 외부 연동 동시 실행량 제한

※ Bulkhead 단독보다 위 3개 조합을 기본안으로 검토한다.

---

## 적용 후보

1. 결제/환불 호출 (`PaymentProvider`)
2. 알림 채널 호출 (`NotificationSender`)
3. 관리자 재시도 API (`/admin/refunds/{id}/retry`)

---

## 적용 트리거

아래 중 하나라도 충족 시 도입을 시작한다.

1. 실제 PG/알림 외부 HTTP 연동이 붙는 시점
2. 운영에서 외부 연동 지연으로 응답 지연/타임아웃이 관측되는 시점
3. 장애 훈련에서 외부 의존성 실패가 전체 API 처리량에 영향 주는 것이 확인된 시점

---

## 현재 결론

지금은 Fake 어댑터 기반이라 즉시 도입 우선순위가 낮다.  
실제 외부 연동을 붙일 때 운영 안정성을 위해 이 항목을 먼저 다시 검토한다.
