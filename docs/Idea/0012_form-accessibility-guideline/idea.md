# 폼 접근성 향상 메모

**날짜**: 2026-03-17  
**상태**: Working note (이미 일부 적용, 이후 유지 기준)

---

## 배경

회원가입, 로그인, 예약, 주문, 관리자 화면처럼 폼 입력이 많은 UI에서는
라벨과 입력 요소의 연결이 끊기면 스크린리더 사용성, 키보드 포커스 이동, 테스트 안정성이 같이 떨어진다.

현재 프론트엔드에는 `Form.Group controlId`를 사용하는 패턴이 넓게 들어가 있고,
이 문서는 그 이유와 이후 유지 기준을 정리한다.

---

## 현재 적용 패턴

- React-Bootstrap `Form.Group controlId`를 사용해 `Form.Label`과 입력 `id`를 연결한다.
- 라벨 클릭 시 해당 입력으로 포커스가 이동하도록 한다.
- 예약/주문/인증/관리자 폼의 주요 입력에는 명시적 `controlId`를 부여한다.

대표 예시:
- 로그인/회원가입 폼
- 예약 생성/조회/변경 폼
- 주문 생성/조회 폼
- 관리자 상품/슬롯/주문 필터 폼

---

## 기대 효과

1. 스크린리더가 라벨과 입력의 관계를 더 정확히 읽는다.
2. 마우스 사용 시 라벨 클릭만으로도 입력 포커스 이동이 가능하다.
3. 키보드 중심 사용자의 흐름이 더 예측 가능해진다.
4. E2E 테스트와 디버깅에서 안정적인 selector 기준이 생긴다.

---

## 유지 원칙

1. 새 폼 입력에는 가능하면 `Form.Group controlId`를 명시한다.
2. placeholder만으로 필드 의미를 전달하지 않고, 설명적인 `Form.Label`을 둔다.
3. 에러 메시지가 있으면 입력과 시각적으로만 아니라 의미상으로도 연결되게 유지한다.
4. 라디오/체크박스처럼 개별 선택 항목은 클릭 영역과 label 연결을 분명히 한다.
5. 접근성은 “스크린리더 배려”뿐 아니라 포커스 이동, 클릭 가능 영역, 테스트 안정성까지 포함한다고 본다.

---

## 아직 남은 점검 포인트

- 에러 메시지와 `aria-describedby` 연결이 필요한 폼이 있는지
- 비활성/로딩 상태가 스크린리더에 충분히 전달되는지
- 모달 내부 폼에서 포커스 트랩과 초기 포커스가 적절한지

---

## 현재 결론

`controlId` 추가는 단순한 스타일 선택이 아니라 접근성 기본선이다.  
앞으로도 새 폼을 만들 때는 라벨-입력 연결을 기본 규칙으로 유지하고, 필요하면 `aria-*` 속성까지 확장한다.
