# 테스트 전략 및 assertion 작성 규칙 메모

**날짜**: 2026-03-17  
**상태**: Working note (PRD 외부 관리)

---

## 배경

`docs/PRD/0001_spec/spec.md`는 제품 요구사항과 현재 운영 계약을 남기는 문서다.  
반면 테스트 철학, 최소 테스트 세트, `SoftAssertions.assertSoftly` 같은 assertion 작성 규칙은 팀 내부 엔지니어링 가이드에 가깝다.

따라서 해당 내용은 PRD에서 분리해 이 문서에서 관리한다.

---

## 테스트 철학

- 테스트는 커버리지 수치보다 비즈니스 검증과 문서화 가치에 집중한다.
- 전체를 다 테스트하기보다 주요 사용/변경 지점을 우선 검증한다.
- 시나리오가 폭증하는 영역은 적은 양으로 여러 계층을 덮는 실용적인 통합 테스트를 우선한다.

---

## 테스트 분류

### 1. 유스케이스 테스트

- 주문 결제 → 승인대기 → 승인/거절/자동환불
- 픽업 마감 자동취소
- 슬롯 정원 8명 동시 예약
- 번들 결제 전체 취소
- 8회권 결제/만료 7일 전 알림/만료 소멸/정산 환불

### 2. 도메인 정책 테스트

- D-1 00:00 환불 경계 판정 (Asia/Seoul)
- 당일 1시간 전 변경 가능 판정
- 제작 시작 이후 환불 불가 전환
- 잔여 크레딧 정산 환불 계산

### 3. 계약 테스트

- 예약/주문 응답 스키마가 문서와 일치하는지
- 알림 페이로드(카카오/SMS) 구성 규칙이 깨지지 않는지

---

## 작성하지 않기로 한 것

- 단순 getter/setter, 단순 매핑, 프레임워크 동작만 확인하는 테스트
- 수동 검증이 더 효율적인 단기 이벤트성 기능 테스트
- 핵심 리스크와 무관한 화면 단순 렌더링 수준 테스트

---

## 핵심 리스크 기반 최소 세트

- 동시성: 단일 재고 동시 주문 충돌, 슬롯 정원 초과 방지
- 상태 멱등성: 승인/거절 중복 호출에도 상태 1회 전이 보장
- 시간 경계: D-1 00:00, 당일 1시간 전
- 자동 배치: 승인 SLA 24시간 자동환불, 픽업 마감 자동취소, 8회권 만료 7일 전 알림/만료 소멸

---

## assertion 및 비교 규칙

- 테스트 메서드에는 `@DisplayName`을 사용하고, 의도를 설명하는 한글 문장으로 작성한다.
- 한 테스트에서 검증 포인트가 2개 이상이면 `SoftAssertions.assertSoftly`를 기본으로 사용한다.
- nullable 가능성이 있는 객체 비교(`String`, `Long`, 토큰/ID 등)는 `Objects.equals(a, b)`를 기본으로 사용한다.
- nullable 객체 비교에서 `a != null && a.equals(b)` 형태의 수동 null 체크 패턴은 새 코드에 사용하지 않는다.

---

## 현재 결론

PRD에는 제품 계약과 운영 정책만 남기고, 테스트 전략과 assertion 작성 규칙은 이 문서에서 계속 관리한다.
