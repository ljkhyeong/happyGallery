# U4. Deferred Verification And Checkout

이 단위는 "휴대폰 인증을 첫 화면에서 요구"하던 흐름을 제출 직전 인증으로 옮기는 작업이다.

성격:
- 프론트 주도
- `U1`, `U2` 계약 확정 후 착수

---

## 1. 목표

- 예약, 8회권 구매, guest 주문에서 탐색과 인증을 분리한다.
- 사용자가 원하는 클래스/시간/상품을 먼저 고른 뒤 마지막 단계에서 로그인/회원가입/비회원 진행을 선택하게 한다.

---

## 2. 범위

포함:
- 예약 생성 플로우 재구성
- 8회권 구매 플로우 재구성
- guest 주문 제출 직전 인증 게이트
- 공통 인증 선택 모달 또는 패널

제외:
- 회원 마이페이지
- guest 이력 claim

---

## 3. 공통 UX 규칙

- 첫 화면은 탐색만 수행한다.
- 제출 버튼 클릭 시 인증 게이트를 연다.
- 인증 게이트 옵션:
  - 로그인
  - 회원가입
  - 비회원 진행
- 회원이면 현재 세션을 사용하고 추가 입력을 최소화한다.
- 비회원 진행을 선택한 경우에만 휴대폰 인증 step을 연다.

---

## 4. 주요 작업

예약:
- `BookingCreatePage`에서 `PhoneVerificationStep` 선행 제거
- `SlotSelectionStep` → 예약자 정보/결제 정보 → 제출 직전 인증 게이트

8회권:
- `PassPurchasePage`에서 `PhoneVerificationStep` 선행 제거
- 권종/안내/금액 확인 후 인증 게이트

주문:
- 상품 상세 구매 버튼 또는 `/orders/new` 제출 직전 인증 게이트

공통:
- 인증 선택 모달/드로어
- guest 휴대폰 인증 step 재사용
- 회원 로그인/회원가입 성공 후 원래 흐름으로 복귀

---

## 5. 완료 기준

- 예약/8회권/주문 첫 진입에서 휴대폰 인증이 바로 뜨지 않는다.
- 제출 직전에만 인증이 필요하다.
- 회원과 비회원 분기 UX가 동일한 공통 패턴으로 보인다.

---

## 6. 최소 검증

- `cd frontend && npm run build`
- 예약/8회권/주문 happy-path 수동 점검 또는 E2E
