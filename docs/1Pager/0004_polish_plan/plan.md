# Post-MVP Polish & Hardening Plan

MVP 기능 연결이 끝난 뒤, 서비스를 “보여줄 수 있는 상태”에서 “실사용 가능한 상태”로 끌어올리기 위한 후속 정리 플랜이다.  
문서 동기화, UX 개선, 관리자 운영성 보강, 인증/관측성, E2E 검증을 단위별로 나눠 진행한다.

---

## ✨ 한눈에 보기

- 먼저 할 일: `P1 -> P2 -> P3`
- 운영 보강: `P5 -> P6 -> P7 -> P4`
- 배포 전 마무리: `P8 -> P9 -> P10`
- 병렬 가능:
  - `P1`, `P2`
  - `P5`, `P6`, `P7`
- 기본 원칙: 한 번에 `단위 1개`만 진행하고, 구현과 문서를 항상 같이 갱신한다.

## 🛠️ 작업 원칙

- 한 번의 지시는 `단위 1개`만 선택한다.
- API 계약, 라우팅, 운영 방식이 바뀌면 관련 문서를 함께 갱신한다.
- 테스트는 변경 범위에 맞는 최소 검증부터 선택한다.
- 프론트 전용 변경은 기본적으로 `cd frontend && npm run build`를 통과해야 한다.
- 백엔드 API/보안/운영 변경은 관련 Gradle 검증이 통과해야 완료로 본다.
- 단위 간 의존성이 있으면 아래 권장 순서를 우선 따른다.

## P1. 스펙 문서 동기화

### 범위

- `docs/PRD/0001_spec/spec.md`
- `docs/1Pager/0003_frontend_plan/api-gap-analysis.md`

### 작업 목표

- B2에서 추가한 공개 조회 API(`products`, `classes`, `slots`)를 실제 컨트롤러 기준으로 정리한다.
- B3의 `passes/purchase`, B4의 공개 주문 생성/조회 계약을 스펙 문서에 반영한다.
- 프론트 라우팅 구조와 공개 진입 경로를 문서에 기록한다.
- `api-gap-analysis.md`에서 이미 해소된 GAP 상태를 최신 구현 기준으로 갱신한다.

### 완료 조건

- 스펙 문서의 API 섹션이 실제 컨트롤러/DTO와 1:1로 대응한다.
- 프론트 기준 문서에서 이미 해소된 공백이 더 이상 “미구현”으로 남아 있지 않다.

### 최소 검증 명령

- 문서 작업만 수행

## P3. 폼 검증 및 에러 UX 강화

### 범위

- `frontend/src/features/*/`
- `frontend/src/shared/ui/ErrorAlert.tsx`
- 필요 시 `frontend/src/shared/api/*`

### 작업 목표

- 클라이언트 측 입력 검증을 공통 기준으로 보강한다.
  - 전화번호 포맷
  - 필수 필드
  - 수량/금액 범위
- `ErrorCode`별 사용자 친화 메시지 매핑을 추가한다.
- 네트워크 오류, 타임아웃, 서버 오류를 구분해 안내한다.
- 제출 중 중복 클릭과 로딩 상태 표시를 일관되게 맞춘다.

### 완료 조건

- 잘못된 입력은 서버 요청 전에 안내된다.
- 주요 에러가 사용자 관점의 메시지로 일관되게 노출된다.

### 최소 검증 명령

- `cd frontend && npm run build`

## P8. E2E 시나리오 검증

### 범위

- Docker Compose 환경
- 백엔드 로컬 실행
- 프론트 개발 서버

### 작업 목표

- 핵심 사용자 플로우를 실제 실행 환경에서 수동 검증한다.
- 발견된 버그는 별도 단위로 분리하거나 같은 작업 안에서 즉시 수정한다.
- 최소 검증 시나리오:
  1. 상품 등록 → 목록 확인
  2. 클래스/슬롯 생성 → 예약 생성 → 예약 조회/변경/취소
  3. 8회권 구매 → 8회권으로 예약
  4. 주문 생성 → 관리자 승인 → 픽업 준비 → 픽업 완료
  5. 환불 실패 → 재시도

### 완료 조건

- 핵심 5개 시나리오가 에러 없이 완주된다.

### 최소 검증 명령

- `docker compose up -d`
- `./gradlew :app:bootRun`
- `cd frontend && npm run dev`

## P9. 프로덕션 인증 계층

### 범위

- 백엔드: Spring Security 또는 필터 기반 인증
- `AdminAuthFilter`
- 프론트: 관리자 로그인 화면

### 작업 목표

- 관리자 인증을 개발용 API Key 수준에서 운영 가능한 인증 계층으로 격상한다.
- 최소 후보:
  - 사용자명/비밀번호 기반 세션
  - JWT 기반 로그인
- `X-Admin-Key`는 개발/내부용 fallback으로만 남기고 운영 비활성화 옵션을 둔다.

### 완료 조건

- 관리자 API가 인증 없이 호출되지 않는다.
- 프론트 관리자 화면에 로그인 흐름이 있다.

### 최소 검증 명령

- `./gradlew :app:test`
- `cd frontend && npm run build`

## P10. 관측성 및 운영 준비

### 범위

- `application.yml`
- `RequestIdFilter`
