# ADR-0027: 테스트 전략 및 최소 테스트 세트 기준선

**날짜**: 2026-03-21
**상태**: Accepted

---

## 컨텍스트

테스트와 문서는 모두 코드와 함께 장기적으로 유지·수정·삭제 비용이 드는 관리 자산이다.
시스템이 커질수록 이 자산의 절대량이 늘어나고,
그만큼 변경 시 동기화 비용과 drift 위험도 함께 커진다.

따라서 테스트 전략은 "많이 작성하는 것"보다
"작성 가치가 높은 테스트만 남기는 것"을 기본 원칙으로 삼아야 한다.

특히 대부분의 시스템에서는 대략 상위 20% 안팎의 핵심 기능이
실제 사용과 변경의 대부분을 차지한다.
테스트도 이 구간에 집중할 때 투자 대비 효과가 가장 크다.

또한 커버리지 수치 향상만을 목적으로 추가된 테스트는
비즈니스 검증 가치가 낮고, 변경 비용만 키우는 경우가 많다.
따라서 테스트는 커버리지 지표보다 비즈니스 검증과 문서화 가치를 우선해야 한다.

---

## 결정 사항

### 1. 테스트의 기본 목적은 비즈니스 검증과 문서화다

- 테스트는 커버리지 수치 향상 자체를 목표로 삼지 않는다.
- 테스트는 핵심 규칙, 상태 전이, 시간 경계, 운영 계약을 자동 검증하고 문서화하는 수단으로 본다.
- 존재 이유가 약한 테스트는 작성하지 않거나 제거한다.

### 2. 테스트와 문서의 절대량 자체를 최소화한다

- 테스트와 문서 모두 시스템과 함께 성장하는 관리 대상이다.
- 새로운 테스트를 추가할 때는 "지금 이 테스트가 장기 유지 비용을 정당화하는가"를 먼저 본다.
- 동일한 위험을 여러 계층에서 중복 검증하는 패턴은 기본값으로 두지 않는다.
- 단순 수치 목표 때문에 필요하지 않은 테스트를 추가하지 않는다.

### 3. 테스트는 핵심 사용/변경 지점에 우선 배치한다

- 일반적으로 상위 20% 안팎의 핵심 기능이 주요 사용과 변경을 차지한다고 가정한다.
- 테스트는 이 핵심 경로에 우선 투자한다.
- 우선순위 예시는 다음과 같다.
  - 주문 결제부터 승인/거절/자동환불까지의 상태 전이
  - 예약 생성/변경/취소와 슬롯 정원 경합
  - 8회권 결제/차감/환불/만료
  - 픽업 만료 자동환불, 배치 처리, 주요 운영 필터/인증 경로

### 4. 테스트 분류는 세 가지를 기본 축으로 한다

#### 유스케이스 테스트

- 여러 계층을 함께 거치며 핵심 흐름이 실제 운영과 유사하게 성립하는지 검증한다.
- 기본 엔트리포인트는 `@UseCaseIT`다.
- 적용 대상:
  - 핵심 상태 전이
  - 동시성 충돌
  - 시간 경계가 걸린 흐름
  - 배치/필터/세션/레이트리밋처럼 실제 인프라 경계가 중요한 흐름

#### 도메인 정책 테스트

- 순수 비즈니스 규칙과 경계 조건을 빠르게 검증한다.
- `policyTest` 범위에서 유지한다.
- 적용 대상:
  - D-1 00:00 환불 경계
  - 당일 1시간 전 변경 가능 여부
  - 제작 시작 이후 환불 불가
  - 8회권 만료/잔여 크레딧/정산 계산

#### 직렬화/역직렬화 테스트

- 외부 계약으로 취급되는 응답/요청/알림 payload가 깨지지 않는지 검증한다.
- 응답 스키마, 에러 응답 포맷, 알림 페이로드처럼 계약 가치가 높은 경우에만 작성한다.
- 단순 DTO 매핑이나 내부 전용 구조는 기본적으로 대상에서 제외한다.

### 5. 다음 종류의 테스트는 기본적으로 작성하지 않는다

- 단순 getter/setter 검증
- 단순 매핑 결과만 확인하는 테스트
- 프레임워크 자체 동작을 반복 확인하는 테스트
- 핵심 리스크와 무관한 화면 단순 렌더링 수준 테스트
- 수동 검증이 더 효율적인 단기 이벤트성 기능 테스트

### 6. assertion 및 작성 규칙은 간결하게 유지한다

- 테스트 메서드에는 `@DisplayName`을 사용하고 한글 문장으로 의도를 드러낸다.
- 검증 포인트가 2개 이상이면 `SoftAssertions.assertSoftly`를 기본으로 사용한다.
- nullable 객체 비교는 `Objects.equals(a, b)`를 기본으로 한다.
- 수동 null 체크 패턴은 새 테스트에서 기본값으로 사용하지 않는다.

## 결과

### 장점

- 테스트와 문서의 총량이 불필요하게 커지는 것을 막을 수 있다.
- 핵심 비즈니스 흐름에 테스트 투자를 집중할 수 있다.
- 커버리지 지표보다 실제 변경 리스크를 더 잘 반영한다.

### 단점

- 주변 기능의 회귀는 의도적으로 덜 자동화될 수 있다.
- 무엇이 "핵심 20%"인지에 대한 판단이 팀 합의에 의존한다.
- 최소 세트 원칙을 잘못 적용하면 필요한 테스트까지 빠질 수 있다.

### 대응

- 신규 기능 추가 시 "핵심 사용/변경 지점인지"를 먼저 판별한다.
- 계약/정책/유스케이스 중 어느 축에서 검증할지 한 곳만 우선 선택한다.
- 중복 테스트가 늘면 추가보다 정리/삭제를 먼저 검토한다.
