# ADR-0016: API URI 버저닝 전략 도입

**날짜**: 2026-03-06  
**상태**: Accepted

---

## 컨텍스트

현재 API 경로는 무버전(`/bookings`, `/passes`, `/products`, `/admin/**`)으로 운영되고 있다.
이 상태에서 요청/응답 계약의 브레이킹 변경이 발생하면 구버전 앱/클라이언트가 즉시 장애를 일으킬 수 있다.

또한 버저닝 규칙이 없는 상태에서 ad-hoc 경로(`v1_new` 등)가 생기면 운영/문서/테스트 일관성이 무너진다.

---

## 결정 사항

### 1. URI Versioning을 표준으로 채택한다

- 표준 API 경로: `/api/v1/**`
- 신규 기능, 문서, 테스트는 `/api/v1/**`를 기준으로 작성한다.

### 2. 무버전 경로는 한시적으로 병행 제공한다

- 기존 경로(`/bookings`, `/passes`, `/products`, `/admin/**`)는 구버전 클라이언트 호환을 위해 유지한다.
- 컨트롤러는 클래스 레벨 `@RequestMapping`에 v1/legacy 경로를 함께 선언한다.

### 3. 브레이킹 변경은 `/api/v2/**`로 분리한다

- `v1` 계약을 깨는 변경은 `v2` 엔드포인트로 신규 제공한다.
- `v1`은 사전 공지된 deprecate 기간 후 제거한다.

---

## 결과 (트레이드오프)

| 항목 | 내용 |
|------|------|
| 장점 | 구버전 클라이언트 호환을 유지하면서 버전 체계를 도입할 수 있다 |
| 장점 | 운영/문서/테스트 기준 경로를 `/api/v1/**`로 통일할 수 있다 |
| 단점 | 전환 기간 동안 동일 기능 경로가 2개(버전/레거시)로 유지된다 |
| 대응 | PRD/ADR에 표준 경로와 폐기 정책을 명시하고, 단계적으로 레거시 제거 |

---

## 구현 반영

- 컨트롤러 9개에 `@RequestMapping({"/api/v1/...", "/..."})` 형태로 반영
- `docs/PRD/0001_기준_스펙/spec.md`에 API 버전 정책(11-A) 및 v1 엔드포인트 예시 반영
