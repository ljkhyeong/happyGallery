# ADR-0027: 테스트 우선순위와 최소 테스트 세트

**날짜**: 2026-03-21
**상태**: Accepted

---

## 왜 이 문서가 필요한가

테스트는 많다고 좋은 것이 아니다.  
변경 비용이 큰 시스템일수록, 어떤 테스트를 남기고 어떤 테스트를 만들지 않는지가 더 중요하다.

이 문서는 happyGallery에서 테스트를 어디에 우선 투자할지 정리한다.

---

## 결정

### 1. 테스트의 목적은 비즈니스 검증과 문서화다

- 커버리지 수치 자체를 목표로 삼지 않는다.
- 핵심 규칙, 상태 전이, 시간 경계, 운영 계약을 자동 검증하는 데 집중한다.
- 존재 이유가 약한 테스트는 만들지 않거나 제거한다.

### 2. 테스트는 핵심 사용 흐름과 변경이 잦은 지점에 먼저 둔다

우선순위 예시:

- 주문 결제 -> 승인/거절/자동 환불
- 예약 생성/변경/취소와 슬롯 정원 경합
- 8회권 결제/차감/환불/만료
- 픽업 만료 자동 환불, 배치 처리, 주요 인증/필터 경로

### 3. 테스트는 세 가지 축으로 나눈다

#### 유스케이스 테스트

- 여러 계층을 함께 거치며 핵심 흐름이 실제처럼 동작하는지 본다.
- 기본 진입점은 `@UseCaseIT`

#### 도메인 정책 테스트

- 순수 규칙과 경계 조건을 빠르게 검증한다.
- `policyTest` 범위에서 유지한다.

#### 직렬화/역직렬화 테스트

- 응답 스키마, 에러 포맷, 알림 payload처럼 외부 계약 가치가 큰 경우만 작성한다.

### 4. 다음 테스트는 기본값으로 만들지 않는다

- 단순 getter/setter 검증
- 단순 DTO 매핑 확인
- 프레임워크 자체 동작을 반복 확인하는 테스트
- 핵심 리스크와 무관한 단순 화면 렌더링 테스트

### 5. 작성 규칙은 간결하게 유지한다

- 테스트 메서드는 `@DisplayName` 한글 문장을 사용한다.
- 검증 포인트가 여러 개면 `SoftAssertions.assertSoftly`를 우선 사용한다.
- 같은 위험을 여러 계층에서 중복 검증하지 않는다.

---

## 기존 문서와의 관계

- 통합 테스트 인프라와 `@UseCaseIT`는 [ADR-0026](../0026_통합_테스트_프로파일과_TestContainer_기준선/adr.md)에서 관리한다.
- 이 문서는 무엇을 왜 테스트할지에 집중한다.

---

## 결과

### 장점

- 테스트 총량이 불필요하게 늘어나는 것을 막을 수 있다.
- 핵심 흐름에 검증을 집중할 수 있다.
- 커버리지 숫자보다 실제 변경 리스크를 더 잘 반영한다.

### 단점

- 주변 기능의 회귀는 덜 자동화될 수 있다.
- 무엇이 핵심인지에 대한 판단이 필요하다.

### 대응

- 신규 기능마다 핵심 사용 흐름인지 먼저 판단한다.
- 계약, 정책, 유스케이스 중 한 축을 먼저 선택한다.
- 중복 테스트가 늘면 추가보다 정리와 삭제를 먼저 검토한다.
