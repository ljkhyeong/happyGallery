# ADR-0019: 비밀번호 해시 정책 (Salt + Key Stretching)

**날짜**: 2026-03-06  
**상태**: Accepted

---

## 컨텍스트

사용자 비밀번호 저장 시 단순 해시(SHA-256, MD5)를 그대로 사용하면
오프라인 크래킹 및 사전(Rainbow Table) 공격에 취약하다.

본 프로젝트는 현재 관리자 인증/인가 체계를 확장 중이며,
사용자 계정 비밀번호 정책을 사전에 고정해 구현 편차를 방지할 필요가 있다.

---

## 결정 사항

### 1. 단순 해시 저장을 금지한다

- SHA-256, MD5 등 단순 해시 단독 사용 금지
- 커스텀 해시 포맷(직접 Salt/반복 구현)도 기본 정책에서 제외

### 2. Salt + Key Stretching 기반 알고리즘만 허용한다

- 허용 알고리즘: `BCrypt`, `Argon2id`, `Scrypt`
- 기본 구현 표준:
  - Spring Security `PasswordEncoder` 사용
  - 기본 권장 구현체는 `BCryptPasswordEncoder`

### 3. 저장/운영 원칙을 고정한다

- DB에는 평문 비밀번호를 저장하지 않는다
- 운영 로그에는 비밀번호 및 해시 원문을 남기지 않는다
- 해시 비교는 반드시 `PasswordEncoder.matches(...)`를 사용한다

## 결과 (트레이드오프)

| 항목 | 내용 |
|------|------|
| 장점 | 크래킹 난이도 상승 및 단순 해시 기반 취약점 제거 |
| 장점 | Spring Security 표준 사용으로 구현 일관성 확보 |
| 단점 | 인증 시 CPU 비용 증가 |
| 대응 | 로그인/인증 엔드포인트에 Rate Limiting 적용, 필요 시 해시 강도(cost) 단계적 조정 |
