# 비회원/회원 흐름 수렴 회고

**날짜**: 2026-03-17  
**상태**: 회고 메모

---

## 배경

`main` 히스토리를 보면 공개 비회원 흐름이 먼저 빠르게 자리잡고,
그 위에 회원 스토어, `/my`, 비회원 이력 가져오기(claim)가 덧붙는 방식으로 기능이 확장됐다.

대표 구간:
- 예약 생성/변경/취소: `052f727`, `26eda11`, `cb5e1d7`
- 주문/패스/상품/주문 승인: `312870d`, `faa4333`, `523719d`, `0ea5c36`
- 회원 스토어/셀프서비스/이력 가져오기(claim): `81aaa36`, `3ca53dd`, `ca4f18e`, `6c4321e`, `835589c`

이 흐름은 기능 확장에는 유효했지만,
이제는 비회원, 회원, 이전 완료 상태를 따로 다루는 중복 비용이 커졌다.

---

## 관찰

- 비회원 예약/주문/8회권을 먼저 구현한 뒤 회원 흐름이 추가되면서
  검증, 응답 조합, 조회 모델이 서로 다른 경로에 중복됐다.
- 비회원 이력 가져오기(claim)가 들어오면서 같은 예약/주문 데이터를 비회원 시점과 회원 시점에 각각 다르게 읽는 요구가 생겼다.
- 최근 비회원 토큰 강화, 예약 리마인더, 관리자 조회 보강 같은 작업도
  결국 “비회원 전제 모델”을 걷어내는 쪽으로 흘렀다.

---

## 회고 포인트

### 1. 예약의 고객 표현을 공통 모델로 정리

현재:
- 예약 조회와 운영 기능이 여전히 비회원 여부 분기를 품기 쉽다.

정리 방향:
- `customerSummary` 또는 `bookingRecipient` 같은 공통 조회 모델을 두고
  비회원, 회원, 이전 완료 상태를 한 모델로 다룬다.

### 2. 예약 생성 흐름을 비회원/회원 공통 코어로 정리

현재:
- 슬롯 활성 검증, 중복 예약 방지, 예약 확정, 이용권/예약금 분기, 이력/알림이
  비회원/회원 서비스에 반복된다.

정리 방향:
- 사용자 식별만 분기하고 나머지는 공통 생성 유틸이나 유스케이스 핵심 흐름으로 묶는다.

### 3. 비회원 이력 이전 이후 운영 경로를 1급 시나리오로 취급

현재:
- 비회원 이력 가져오기(claim)는 회원 전환 기능으로 추가됐지만,
  관리자 조회, 배치, 알림은 비회원 기준 가정을 놓치기 쉽다.

정리 방향:
- 새 예약/주문 기능을 만들 때부터 비회원, 회원, 이전 완료 3상태를 체크리스트로 검토한다.

### 4. 비회원 토큰 후속은 기존 0005와 연결

현재:
- 비회원 흐름 수렴 논의와 토큰 전달/만료 논의가 서로 얽혀 있다.

정리 방향:
- 토큰 자체는 [0005_비회원_토큰_Signed_만료](../../Idea/0005_비회원_토큰_Signed_만료/idea.md)에서 관리하고,
  이 문서는 비회원/회원 흐름 수렴 자체에 집중한다.

---

## 현재 결론

비회원 흐름 위에 회원 흐름을 올리는 방식은 빠른 구현에는 유효했다.  
다만 이후 비용을 줄이려면 “비회원, 회원, 이전 완료 상태를 함께 담는 공통 모델”을 더 일찍 상정했어야 한다.
