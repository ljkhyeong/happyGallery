# 프론트 E2E 실행 시간 슬림화 회고

**날짜**: 2026-04-26
**상태**: 회고 메모

---

## 배경

결제, 예약, 8회권, 비회원 이력 가져오기 흐름을 P8 Playwright 시나리오로 묶으면서 브라우저 E2E의 검증 가치는 커졌다.
동시에 기본 실행 시간이 길어져서 로컬 반복 검증에는 부담이 생겼다.

특히 결제나 예약 흐름을 보려는 테스트가 매번 관리자 UI로 상품과 슬롯을 만들면, 실제로 검증하려는 사용자 경로보다 사전 준비 시간이 더 커진다.
이 상태에서는 브라우저 E2E가 수동 브라우저 확인보다 느리게 느껴지고, 개발 중 자주 실행되지 않는 테스트가 될 위험이 있다.

---

## 관찰

- 기본 E2E 명령이 전체 P8 흐름을 모두 실행하면 빠른 피드백 루프로 쓰기 어렵다.
- 관리자 UI 자체를 검증하지 않는 시나리오에서도 관리자 화면을 통과해 테스트 데이터를 만들고 있었다.
- 결제/예약/회원/관리자 경로는 실패 원인과 실행 목적이 다르므로 같은 기본 명령에 묶기보다 태그로 분리하는 편이 낫다.
- 반복 smoke에서는 백엔드에 `RATE_LIMIT_ENABLED=false`를 주입해 관리자 로그인과 휴대폰 인증 제한이 테스트 순서에 영향을 주지 않게 해야 한다.

---

## 반영한 기준

### 1. 기본 E2E는 smoke로 제한

`frontend npm run e2e`는 `@smoke` 태그의 대표 경로만 실행한다.
기본 명령은 PR 전 로컬 반복 확인에 쓰이는 명령이므로, 전체 회귀보다 빠른 실패 감지가 우선이다.

### 2. 전체 회귀와 도메인 회귀를 명령으로 분리

전체 P8 브라우저 회귀는 `npm run e2e:full`로 실행한다.
결제, 회원/신원, 관리자 경로는 각각 `e2e:payment`, `e2e:identity`, `e2e:admin`으로 나눠 실행한다.

### 3. 사전 데이터는 API fixture 우선

관리자 UI가 검증 대상이 아닌 경우 상품과 슬롯 생성은 API fixture로 처리한다.
관리자 화면의 회귀는 `@admin` 시나리오에 남기고, 결제/예약 사용자 경로의 준비 단계와 섞지 않는다.

---

## 측정 결과

이번 조정 뒤 로컬 실행 결과는 다음과 같다.

- `frontend npm run e2e`: 4개 smoke, 21.8초
- `frontend npm run e2e:full`: 9개 전체 P8, 34.6초

이전 전체 P8 실행은 같은 세션에서 8개 기준 46.9초까지 걸렸다.
테스트 수는 늘었지만 API fixture와 태그 분리로 기본 피드백 시간은 줄었다.

---

## 현재 결론

브라우저 E2E는 수동 확인을 완전히 대체하는 도구가 아니라, 자주 깨지는 표준 사용자 경로를 반복 가능하게 고정하는 도구다.
의미를 유지하려면 기본 실행은 짧아야 하고, 전체 회귀는 필요할 때 명시적으로 선택해야 한다.

추가로 CI에서 smoke와 full/nightly를 분리해야 할 때는 GitHub issue를 만들고, 이 문서와 ADR-0027을 링크한다.
