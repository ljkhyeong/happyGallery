# 로컬/개발 지원 기능 경계 회고

**날짜**: 2026-03-17  
**상태**: 회고 메모

---

## 배경

`main` 히스토리 후반으로 갈수록 로컬 전용 seed, 개발용 실패 주입 API, 테스트 지원 API가 늘어났다.

대표 구간:
- `e009c5d` 프로필 분리
- `fc105cf` admin 인증 필터
- 비회원/회원 E2E 확장 구간의 로컬 보조 코드 반영
- 관측성과 운영 보강 구간의 개발용 지원 기능 증가

최근 보안 정리에서도 기본 관리자 계정, 개발용 API key, OTP 노출을 로컬 경계 안으로 다시 밀어 넣는 작업이 필요했다.

---

## 관찰

- 로컬/개발 편의 기능은 처음에는 유용하다.
  하지만 프로필 경계가 약하면 운영 기본값과 섞이기 쉽다.
- 로컬 전용 컨트롤러/서비스가 기능별로 흩어지면
  무엇이 운영 코드이고 무엇이 지원 도구인지 읽기 어려워진다.
- 테스트와 E2E가 커질수록 개발용 지원 API는 늘어날 가능성이 높다.

---

## 회고 포인트

### 1. 로컬 지원 기능의 명명 규칙을 고정

현재:
- 로컬 seed, 실패 주입 API, 디버그 조회 기능이 기능별로 흩어진다.

정리 방향:
- `Local*Service`, `Local*Controller`, `*DevHook` 같은 명명과 위치 규칙을 고정한다.

### 2. unsafe default는 공통 설정에 두지 않는다

현재:
- 개발 편의 설정이 기본 설정에 들어갔다가 나중에 걷어내는 일이 생긴다.

정리 방향:
- 기본값은 안전하게 두고, 로컬/개발 환경에서만 명시적으로 켠다.

### 3. E2E 지원 API는 운영 기능과 분리해 목록화

현재:
- E2E를 위해 추가한 개발용 endpoint가 점진적으로 늘어난다.

정리 방향:
- 로컬 전용 지원 API를 한곳에 목록화하고,
  README나 운영 문서에는 “운영 경로와 분리된 지원 기능”으로 명확히 적는다.

---

## 현재 결론

로컬/개발 지원 기능은 앞으로도 계속 필요하다.  
문제는 “있느냐 없느냐”가 아니라 “기본 설정과 운영 코드에 얼마나 섞이느냐”다.
처음부터 지원 기능을 별도 경계로 관리하는 규율이 필요하다.
