## 관찰

- 기능 구현 뒤에 README/HANDOFF/PRD/브라우저 테스트 동기화가 후속 작업으로 반복된다.
- 보조 경로와 표준 경로가 한동안 같이 존재하면 운영 카피와 테스트 선택자도 흔들린다.
- 문서 종류별 역할이 늦게 분리될수록 기준 스펙이 무거워진다.

---

## 회고 포인트

### 1. 새 기능 PR에 문서 체크리스트를 붙인다

현재:
- 문서 반영 여부가 PR마다 암묵적으로 처리된다.

정리 방향:
- PR 템플릿이나 작업 규칙에 “README / PRD / API 계약 / 브라우저 테스트 / 운영 문서” 체크 항목을 둔다.

### 2. 공개 경로를 만들 때 표준 경로를 먼저 선언한다

현재:
- 보조 경로와 운영 문구가 뒤늦게 정리된다.

정리 방향:
- 새 공개 경로를 도입할 때 표준 경로, 보조 경로, 제거 시점, 운영 카피 기준을 같이 적는다.

### 3. 문서 분리 기준을 더 일찍 적용한다

현재:
- 기능 설명, 설계 메모, 테스트 가이드가 한 문서에 모이기 쉽다.

정리 방향:
- 요구사항은 PRD, 실험은 POC, backlog는 Idea, 작은 개선은 `simple-idea.md`로 일찍 분리한다.
