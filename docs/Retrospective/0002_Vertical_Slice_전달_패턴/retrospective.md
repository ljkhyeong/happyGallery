# 기능 단위 묶음 개발 방식 회고

**날짜**: 2026-03-17  
**상태**: 회고 메모

---

## 가설

도메인별 기능을 `domain -> infra -> app -> docs/test` 순으로 한 묶음으로 완성하는 방식은
복잡한 공방 운영 기능을 빠르게 쌓는 데 유효하다.

---

## 검증 방법

대표 기능 단위를 기준으로 커밋 흐름을 확인했다.

- 예약 생성: `7a28520`, `a4918c1`, `f6ca34d`, `052f727`, `90cb72f`
- 예약 변경: `699962d`, `469907f`, `2c0db0e`, `26eda11`, `347ded9`, `eb7f9fd`
- 주문/패스/상품/재고/제작/픽업: `312870d`, `faa4333`, `523719d`, `0ea5c36`, `09c2b09`, `966cc48`
- 알림/배치/관리자 인증: `a86bdf0`, `8a7f01e`, `fc105cf`

---

## 결과

- 각 기능 단위는 범위를 자르기 쉬웠고,
  문서와 테스트를 함께 붙이면 요구사항 추적이 비교적 수월했다.
- 빠른 기능 전달에는 효과적이었고,
  실제로 2월 말부터 3월 초까지 코어 운영 기능을 짧은 기간 안에 채울 수 있었다.

한계도 확인됐다.
- 비회원/회원 중복 로직
- 컨트롤러 조합 로직 잔존
- 운영 조회 모델 정리 지연

즉, 기능 단위 전달 방식 자체는 유효하지만,
몇 개의 기능 단위가 쌓인 뒤에는 공통화 점검 시점이 필요하다.

---

## 결론

기능 단위 묶음 개발 방식은 유지할 가치가 있다.  
다만 2~3개 기능 단위가 누적될 때마다 “공통화/경계 정리 단계”를 의도적으로 끼워 넣는 편이 더 낫다.
