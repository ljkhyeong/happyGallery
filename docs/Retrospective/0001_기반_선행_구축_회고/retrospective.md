# 기반 선행 구축 회고

**날짜**: 2026-03-17  
**상태**: 회고 메모

---

## 가설

대형 기능을 붙이기 전에 실행 기반을 먼저 갖추면,
이후 기능 구현과 검증 속도가 눈에 띄게 좋아진다.

---

## 검증 방법

`main` 히스토리에서 아래 기반 커밋과 이후 기능 확장 구간을 비교했다.

대표 커밋:
- `e009c5d` 프로필 분리
- `1316409` spec 작성
- `3635394` docker-compose MySQL
- `9fa3604` Flyway 초기 마이그레이션
- `5ac7655` CI 추가
- `f05bd23` requestId 관측성
- `ab7a6e6` Testcontainers
- `c49bd08`, `c46acaf` Clock/PolicyTest 기반

---

## 결과

- 기반이 잡힌 뒤 기능 커밋은 도메인, 애플리케이션, 인프라, 문서, 테스트를 함께 묶는 기능 단위로 비교적 빠르게 이어졌다.
- 시간 경계, 배치, 동시성, 운영 로그 같은 리스크가
  사후 수작업이 아니라 테스트와 공통 유틸에 실리기 시작했다.
- 이후 예약/패스/상품/주문/알림/배치 구현 속도는
  초기 실험기보다 훨씬 안정적인 형태로 빨라졌다.

---

## 결론

이 저장소에서는 “프로필 + 문서 + DB migration + CI + 테스트 기반”을 먼저 세우는 방식이 실제로 유효했다.

후속:
1. 새 대형 트랙도 최소 실행 기반부터 만든 뒤 기능을 붙인다.
2. 기능 설계와 동시에 최소 검증 전략을 같이 적는다.
