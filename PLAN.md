# Project Plan —
> 목표: spec.md 기준으로 운영 가능한 MVP를 빠르게 만들기  
> 규칙: 각 항목은 “해야 할 일(How)” + “완료 기준(DoD)” + “확인 방법(Proof)”를 포함

---

## 0) 킥오프: 기준선 고정
### 해야 할 일(How)
- [ ] 변경은 PR로만 하도록 규칙을 둔다.
- [ ] 주문/예약/패스의 **상태 목록**(enum 후보)을 문서에 1페이지로 요약한다.
    - 주문: PAID_APPROVAL_PENDING, APPROVED..., REJECTED_REFUNDED, AUTO_REFUNDED_TIMEOUT, ...
    - 예약: BOOKED, CANCELED, NO_SHOW, COMPLETED
    - 패스: ACTIVE, EXPIRED, PARTIAL_REFUNDED(또는 ledger 기반)
- [ ] “범위 외”를 `OUT_OF_SCOPE.md`로 분리하고, 팀/미래의 나를 위해 이유를 1줄씩 적는다.
    - 예: POS 실시간 연동(초기에는 운영 정책으로 흡수), 복잡 쿠폰/적립금(후순위) 등

### DoD
- 스펙/스코프가 고정되어 “무엇을 만들지”가 흔들리지 않음

### Proof(확인)
- 레포에 `spec.md`, `OUT_OF_SCOPE.md` 존재 + 커밋/PR 링크

## 1) 뼈대 세우기(프로젝트 구조/런타임)

### 1.1 레포/모듈 구조
#### 해야 할 일(How)
- [ ] 패키지(또는 모듈) 경계를 미리 만든다(아직 기능 없어도 “칸”만 만든다).
    - catalog, order, booking, pass, payment, notification, admin
    - 공통(common): time, id, error, tx, auth, result(응답 포맷)
- [ ] 실행 프로필을 분리한다.
    - local: 로컬 실행용(DB 포함)
    - prod: 운영용(환경변수 기반)
- [ ] 빌드/실행 루틴을 README에 5줄로 고정한다.

#### DoD
- 빌드/실행 OK + 경계가 생겨서 다음 코드가 들어갈 위치가 확정됨

#### Proof
- `./gradlew build` 성공
- local 프로필로 실행되고 health 확인 가능(다음 단계에서 검증)

---

### 1.2 인프라(로컬)
#### 해야 할 일(How)
- [ ] docker-compose로 MySQL을 띄울 수 있게 한다(초기부터 prod와 DB 맞추기 권장).
- [ ] Flyway(또는 Liquibase)를 선택하고, 마이그레이션이 **앱 시작 시 자동 적용**되게 한다.
- [ ] 기본 관측성(요청ID)을 도입한다.
    - 요청 헤더 `X-Request-Id`가 있으면 사용
    - 없으면 생성
    - 응답에도 동일 헤더로 반환
    - 로그 라인에 requestId가 찍히도록 로그 포맷 반영

#### DoD
- 로컬에서 DB가 뜨고, 앱 시작 시 마이그레이션이 자동 적용됨
- 요청 하나를 기준으로 로그를 “요청 단위”로 추적 가능

#### Proof
- DB에 `flyway_schema_history` 생성 확인
- `/actuator/health` `UP`
- 호출 시 응답 헤더에 `X-Request-Id` 존재 + 로그에 동일 ID 출력

## 2) 데이터 모델 1차 구현(마이그레이션 우선)

### 2.1 핵심 테이블 생성
#### 해야 할 일(How)
- [ ] ERD를 ‘완벽’하게 그리려 하지 말고, **스펙을 구현하는 최소 테이블**을 먼저 만든다.
- [ ] Flyway 마이그레이션으로 아래 테이블을 생성한다.
    - users, guests
    - products, inventory
    - classes, slots
    - bookings, booking_history
    - pass_purchases, pass_ledger
    - orders, order_items, fulfillments, order_approvals, refunds
    - notification_log

#### DoD
- 1개의 마이그레이션(또는 최소 세트)로 전체 핵심 테이블이 생성됨

#### Proof
- 로컬 MySQL에서 테이블 목록이 모두 보임
- 앱 재기동 시 마이그레이션 재적용 없이 정상 부팅

---

### 2.2 인덱스/제약 1차
#### 해야 할 일(How)
- [ ] “동시성/조회” 핵심 지점에만 먼저 인덱스/제약을 넣는다.
    - slots: (class_id, start_at) unique(또는 운영 정책에 맞게)
    - bookings: version 컬럼(낙관적 락 용) 또는 동등한 전략 확정
    - inventory: version/락 전략 반영(단일 작품 중복 판매 방지)
    - orders: (status, approval_deadline_at) 인덱스(승인 SLA 배치 조회용)
    - slots.booked_count 갱신 경로를 명확히(트랜잭션 안에서만 갱신)

#### DoD
- “재고 1개 / 정원 8명” 같은 핵심 동시성 케이스를 DB 구조가 뒷받침함

#### Proof
- 동시 요청 테스트(간단한 수동/스크립트)에서 초과 예약/중복 판매가 구조적으로 막히는 방향이 보임

## 3) “가치 있는 테스트” 프레임 세팅(초기에 박아두기)

### 3.1 테스트 분류 템플릿
#### 해야 할 일(How)
- [ ] 테스트를 목적별로 분리한다.
    - UseCaseIT: 유스케이스 통합 테스트(가치 높은 20%)
    - PolicyTest: 도메인 경계/정책 테스트(시간, 환불, 상태전이)
    - ContractTest: 직렬화/역직렬화/외부 연동 계약(필요할 때만)
- [ ] 통합 테스트 DB 전략을 고른다.
    - Testcontainers(MySQL) 권장(운영 DB와 동일)
    - 또는 H2(속도 우선, 단 차이를 인지)
- [ ] 시간 경계 테스트를 위해 Clock 주입을 공통 유틸로 고정한다(Asia/Seoul).

#### DoD
- 테스트가 “의도”대로 분류되어 있고, 시간 경계 테스트가 가능한 구조

#### Proof
- 정책 테스트 1개 + 통합 테스트 1개가 실행됨

## 4) 공통 컴포넌트(모든 기능의 발판)

### 4.1 시간/경계 규칙 유틸
#### 해야 할 일(How)
- [ ] 아래 규칙을 함수/정책 객체로 만들고 PolicyTest로 검증한다.
    - D-1 00:00(서울 기준) 환불 경계 판정
    - 당일 1시간 전 변경 가능 판정

#### DoD

#### Proof
- D-1 23:59/00:00, 1시간 1분/59분, 90일/7일 전 케이스가 테스트로 통과

---

### 4.2 에러/응답 규격
#### 해야 할 일(How)
- [ ] 에러 코드 체계를 정한다(409/400/404 등).
- [ ] 상태 전이 실패를 표준 예외로 통일한다.
    - AlreadyRefunded(자동환불 이후 승인 시도)
    - CapacityExceeded(정원 초과)
    - InventoryNotEnough(단일 작품 중복)
- [ ] “에러 응답 포맷”을 하나로 고정한다(필드명/구조).

#### DoD
- API에서 에러가 일관된 형태로 떨어짐(클라이언트/운영자 대응 쉬움)

#### Proof
- 409/400 대표 케이스에서 응답 포맷이 동일

#### 해야 할 일(How)

- [ ] 변경 이력은 append-only로 누적한다(booking_history).
- [ ] 동시 변경(2기기/재시도)에도 꼬이지 않게 락/버전 전략을 적용한다.

#### DoD

### 5.4 예약 취소/노쇼/완료

#### Proof

## 6) 결제/환불(예약금 10%) 최소 구현

### 6.1 결제 인터페이스 추상화
#### 해야 할 일(How)
- [ ] PaymentProvider를 추상화하고, 개발용 FakeProvider로 먼저 연결한다.
- [ ] 환불 실패는 “실패 상태로 저장”하고 운영자가 재시도할 수 있어야 한다.

#### DoD
- 결제/환불이 도메인 밖으로 분리되고, 실패가 사라지지 않음

#### Proof

### 6.2 예약금 결제 정책 반영
#### 해야 할 일(How)
- [ ] 예약금은 온라인 PG(카드/간편)만 허용한다(계좌이체 금지).
- [ ] 변경 시 예약금은 유지한다(재결제 없음).
- [ ] D-1 경계에 따른 환불 가능/불가를 강제한다.

#### DoD

#### Proof
- 계좌이체로 예약금 결제 시도는 차단
- 변경 시 추가 결제 없음
